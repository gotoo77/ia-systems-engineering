# Style Guide

## Intention

Stabiliser les règles d'écriture pour garder un ton cohérent dans tout le manuel.

## Rôle du document

Ce fichier est la source de vérité pour les règles d'écriture : ton, structure, vocabulaire et fins de chapitre.

## Règles d'écriture

- Écrire en français clair, technique, sans emphase marketing.
- Commencer par le problème, puis introduire la solution.
- Expliquer le pourquoi avant le comment.
- Définir les responsabilités des composants avant de nommer des produits.
- Employer le même terme pour le même concept ; sinon documenter la nuance dans `GLOSSARY.md`.
- Préférer les phrases courtes et les titres descriptifs.
- Utiliser des analogies seulement si elles réduisent réellement la complexité.
- Limiter les listes d'outils à leur rôle dans une comparaison d'architecture.
- Éviter les formulations dépendantes de l'actualité : "le meilleur outil", "actuellement incontournable", "nouvelle référence".
- Ne pas mélanger tutoriel, opinion et référence conceptuelle dans la même section.

## Structure d'un chapitre

Chaque chapitre doit contenir ou préparer :

- un problème d'architecture ;
- une intention claire ;
- des questions de conception ;
- des principes ;
- des compromis ;
- des limites ou modes d'échec ;
- un exemple minimal si nécessaire ;
- des termes reliés au glossaire ;
- une conclusion courte.

## Fin de chapitre

Chaque chapitre complet doit se terminer par :

- `À retenir`
- `Limites`
- `Questions ouvertes`
- `Pour aller plus loin`

## Questions à traiter

- Le texte explique-t-il pourquoi le sujet compte ?
- Le lecteur repart-il avec un modèle mental durable ?
- Quelle conclusion abusive faut-il empêcher ?
